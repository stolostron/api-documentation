**Date:** 2026-04-08  
**Branch:** release-2.16  
**Status:** Approved — implementing

## Context

This repo (`api-documentation`) downloads CRDs from three `stolostron` GitHub repos and generates Markdown + JSON API docs via `cmd/gen-api-docs.py`. CRD `openAPIV3Schema.description` fields flow **unchanged** to output — the generator does not augment them. Description quality directly affects the usefulness of both human docs and AI-consumable schemas.

**Problem:** No systematic quality tracking exists. CRD source repos are not stored here; real CRDs require `make setup-core`. The `tests/test_data/` CRDs are synthetic unit-test fixtures — not representative of production quality.

**Goal:** Build grading infrastructure — a dated report grading real ACM/MCE CRDs, plus reusable commands for Claude Code and Gemini CLI to repeat grading each release.

## Grading Rubric

| Grade | Criteria | Example |
|---|---|---|
| **Good** | Explains what AND why/when to use it; units for numeric fields; enum values referenced; non-redundant | "Timeout in seconds before the reconciliation attempt is aborted" |
| **Passable** | Correct but minimal; missing units/enum context or slightly redundant | "Number of retries to run" |
| **Needs Improvement** | Absent, repeats field name, too vague, or misleading | "Configuration object", "retries", *(absent)* |

## Grading Workflow

```
# Step 1: Download real CRDs
make setup-core

# Step 2a: Grade with Claude Code
/grade-crds

# Step 2b: Grade with Gemini CLI
gemini -m gemini-3-fast < prompts/grade-crd-descriptions.md
```

Both tools write to `grading/crd-descriptions-YYYY-MM-DD.md`.

---

## `/grade-crds` Command Logic

1. **Locate CRDs** — prefer `acm-crds/` and `mce-crds/`; fall back to `tests/test_data/` with a note
2. **Extract descriptions** — walk `openAPIV3Schema` recursively; record field path + description (or "absent")
3. **Apply rubric** — grade each field Good / Passable / Needs Improvement
4. **Write report** — one table per CRD; aggregate summary; cross-CRD patterns; top 5 recommendations
5. **Report back** — tell user the path, summary counts, and worst 3–5 fields

### Top 5 Fields Most in Need of Improvement

1. **`Search.spec` / `Search` (root)** — Ships literal `[PLACEHOLDER, NOT IMPLEMENTED]` strings in production. Content-free boilerplate.
2. **`Agent.spec.approved` (boolean, absent)** — No description. Most operationally critical field in the Agent CRD; controls whether a host is admitted into cluster installation.
3. **`UserPreference` (entire CRD)** — Zero `description:` fields anywhere in the schema. Docs only exist as YAML comments, invisible to `kubectl explain` and doc generators.
4. **`Restore.status.velero*RestoreName` (4 fields)** — All four velero status fields missing descriptions.
5. **`ClusterInstance.spec.clusterType` (enum)** — Says only "a string representing the cluster type" despite four distinct enum values (SNO, HighlyAvailable, HostedControlPlane, HighlyAvailableArbiter) with different operational implications.
